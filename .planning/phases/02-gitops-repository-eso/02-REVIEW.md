## Summary

Reviewed the Phase 2 GitOps Bridge implementation spanning both the infra-setup Terraform repository (oci-vault extension, oci-argocd-bootstrap module, root wiring) and the gitops-setup repository (ApplicationSet, addon Applications, ESO manifests, values hierarchy). The Terraform changes are well-structured with proper sensitive variable handling, lifecycle protection, and clean module wiring. The GitOps manifests follow correct ESO v1 API syntax and ArgoCD multi-source patterns. One critical issue found with the argocd-bootstrap module violating the infra module convention by including its own `versions.tf`. Several warnings around sync ordering and hardcoded values that reduce the GitOps Bridge pattern's flexibility.

## Critical Issues

### CR-01: Infra module `oci-argocd-bootstrap` has its own `versions.tf` -- violates project convention

**File:** `terraform/infra/modules/oci-argocd-bootstrap/versions.tf:1-16`
**Issue:** The project convention (documented in CLAUDE.md) states: "Infra modules (`terraform/infra/modules/`) do NOT have `versions.tf` files -- providers are inherited from the root." The new `oci-argocd-bootstrap` module declares its own `required_providers` block with `helm`, `kubernetes`, and `kubectl` providers. No other infra module has a `versions.tf`. This breaks the established convention and could cause provider version conflicts if the root and module constraints diverge.
**Fix:** Delete `terraform/infra/modules/oci-argocd-bootstrap/versions.tf` entirely. The module will inherit providers from the root `terraform/infra/versions.tf` which already declares all three providers (`helm ~> 3.0`, `kubernetes ~> 3.0`, `kubectl ~> 2.0`).

## Warnings

### WR-01: ESO CRDs may not exist when ClusterSecretStore and ExternalSecrets are applied during bootstrap

**File:** `/home/rodrigo/projects/AssessForge/gitops-setup/bootstrap/control-plane/addons/eso/cluster-secret-store.yaml:9`
**Issue:** The `addon-eso` Application (generated by the AppSet) syncs the `eso/` directory contents. The inner ESO `application.yaml` (sync-wave "1") creates an ArgoCD Application that deploys the ESO Helm chart. The ClusterSecretStore and ExternalSecrets (sync-wave "2") are CRs that require ESO CRDs to exist. However, sync waves only order resource application within a single sync -- they do not wait for child Applications to complete their sync. At wave 2, ArgoCD will attempt to apply the CRs before ESO has finished deploying and registering its CRDs, causing transient sync failures until `selfHeal` retries succeed.
**Fix:** This is acceptable for a bootstrap scenario (retries will eventually converge), but document this expected behavior. Alternatively, consider adding the ESO CRDs as a separate sync-wave "0" resource using a raw CRD manifest or a dedicated CRD-only Helm chart, so CRDs are registered before wave 2. The simplest approach is to accept the transient errors and add a comment in `cluster-secret-store.yaml`:
```yaml
# Nota: durante bootstrap inicial, este recurso pode falhar temporariamente
# ate que o ESO (sync-wave 1) termine de registrar seus CRDs.
# selfHeal garante convergencia automatica.
```

### WR-02: Addon Applications hardcode repo URL and revision instead of using Bridge Secret annotations

**File:** `/home/rodrigo/projects/AssessForge/gitops-setup/bootstrap/control-plane/addons/eso/application.yaml:11-12`
**Issue:** The ESO Application hardcodes `repoURL: 'https://github.com/AssessForge/gitops-setup'` and `targetRevision: 'main'`. The same pattern appears in the ArgoCD Application (`bootstrap/control-plane/argocd/application.yaml:11-12`), Envoy Gateway (`envoy-gateway/application.yaml:11-12`), cert-manager (`cert-manager/application.yaml:11-12`), and metrics-server (`metrics-server/application.yaml:11-12`). The Bridge Secret already carries `addons_repo_url` and `addons_repo_revision` annotations for exactly this purpose (used by the ApplicationSet). Hardcoding these values means changing the repo URL or branch requires editing every Application manifest instead of just updating the Bridge Secret.
**Fix:** Since these are standalone Application manifests (not generated by the AppSet template), they cannot directly reference Bridge Secret annotations. This is a known limitation of the app-of-apps pattern vs. the ApplicationSet pattern. Two options: (1) Accept the hardcoding and document that repo URL changes require updating all Application manifests. (2) Restructure so the ApplicationSet template itself points to the Helm chart sources, eliminating the inner Application manifests. For v1, option 1 is pragmatic -- add a comment at the top of each application.yaml noting this coupling.

### WR-03: `gitops_repo_pat_ocid` output unnecessarily marked as sensitive

**File:** `terraform/infra/modules/oci-vault/outputs.tf:16-20`
**Issue:** The `gitops_repo_pat_ocid` output is marked `sensitive = true`, but OCIDs are resource identifiers, not credentials. They are not secrets -- knowing a Vault secret's OCID does not grant access to the secret's contents (IAM policies control that). Marking it sensitive hides the value in `terraform output` and plan output, adding operational friction when debugging or wiring outputs to other modules, without any security benefit.
**Fix:**
```hcl
output "gitops_repo_pat_ocid" {
  description = "OCID do secret GitHub PAT no Vault"
  value       = oci_vault_secret.gitops_repo_pat.id
}
```

## Info

### IN-01: `PLACEHOLDER_VAULT_OCID` in ClusterSecretStore requires manual replacement

**File:** `/home/rodrigo/projects/AssessForge/gitops-setup/bootstrap/control-plane/addons/eso/cluster-secret-store.yaml:16`
**Issue:** The ClusterSecretStore has `vault: "PLACEHOLDER_VAULT_OCID"` which must be replaced with the actual Vault OCID after `terraform apply`. The comment documents how to get the value (`terraform -chdir=../infra-setup/terraform/infra output vault_ocid`), but there is no automation or validation to catch a forgotten placeholder. The same placeholder exists in `clusters/in-cluster/addons/eso/values.yaml:8` but is unused by any manifest (the ClusterSecretStore hardcodes the value rather than reading from Helm values).
**Fix:** Consider adding a CI check (e.g., `grep -r PLACEHOLDER` in the gitops-setup repo) to catch unreplaced placeholders before merge. Also consider whether the `clusters/in-cluster/addons/eso/values.yaml` placeholder should be consumed by the ClusterSecretStore via Helm templating rather than being hardcoded in the raw YAML manifest, to centralize the configuration point.

### IN-02: Unused `clusters/in-cluster/addons/eso/values.yaml` -- vault_ocid not consumed

**File:** `/home/rodrigo/projects/AssessForge/gitops-setup/clusters/in-cluster/addons/eso/values.yaml:8`
**Issue:** This file defines `vault_ocid: "PLACEHOLDER_VAULT_OCID"` but no manifest or Helm chart references it. The ESO Application in `eso/application.yaml` uses `$values/environments/default/addons/eso/values.yaml` for Helm values (which only sets `installCRDs: true`). The ClusterSecretStore is a raw YAML manifest, not a Helm template, so it cannot consume values from this file. This is dead configuration that could mislead operators into thinking they only need to update this file.
**Fix:** Either remove the `vault_ocid` from this file (keeping it as a stub or removing the file), or restructure so the ClusterSecretStore is templated through the Helm chart's values pipeline. For v1, removing the misleading value and adding a comment is sufficient:
```yaml
# Valores especificos do cluster para ESO
# ClusterSecretStore e ExternalSecrets sao manifestos raw em bootstrap/control-plane/addons/eso/
# O vault_ocid deve ser configurado diretamente em cluster-secret-store.yaml
```
