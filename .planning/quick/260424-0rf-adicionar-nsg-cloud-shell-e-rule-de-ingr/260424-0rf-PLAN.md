<objective>
Add an additive path for the first-apply bootstrap using OCI Cloud Shell Private Network, as a replacement for the Bastion tunnel path that hit a TLS handshake timeout in OCI Bastion's managed port-forwarding.

Purpose: Unblock the Phase 1 → Phase 2 bootstrap today. The existing `bastion-first-apply.md` path is confirmed broken for TLS-heavy kube-apiserver traffic on this tenancy. Cloud Shell Private Network attaches directly to the VCN via a customer-supplied NSG, reaches the private OKE API endpoint at wire speed, and stays inside Always Free tier. Zero cluster changes — no destroy/recreate risk.

Output:
- Terraform: one new NSG (`assessforge-nsg-cloud-shell`), one new ingress rule on the `api_endpoint` NSG (TCP 6443 from the new NSG), one module output, one root-level output passthrough.
- Docs: one new runbook `docs/runbooks/cloud-shell-first-apply.md` covering the end-to-end procedure.

Non-goals:
- Do NOT modify or delete `docs/runbooks/bastion-first-apply.md` — keep it as a still-valid option for operators with shells that can handle the TLS fragmentation.
- Do NOT modify `scripts/bastion-first-apply.sh`.
- Do NOT flip the OKE endpoint to public — blocked by ForceNew on `endpoint_config.subnet_id` and by the `prevent_destroy = true` on the cluster.
</objective>

<background>
Operator attempted the Bastion tunnel bootstrap on 2026-04-24 and reached TCP connectivity over the tunnel, but the TLS handshake to kube-apiserver stalled (fragmented ClientHello dropped by OCI Bastion's managed port-forwarding path). Confirmed not an SNI issue, not a WSL2 `ssh -f` issue — it is a documented behavior of OCI Bastion for TLS-heavy workloads.

Alternatives considered:
- Flip OKE endpoint to public → BLOCKED: `endpoint_config.subnet_id` is ForceNew, cluster has `prevent_destroy = true`.
- Compute VM jump host → FORBIDDEN by operator (ARM Always Free quota already consumed by node pool; paid shape would violate CLAUDE.md constraint).
- **OCI Cloud Shell Private Network → CHOSEN**: attaches Cloud Shell to a VCN subnet with a customer-supplied NSG, reaches private endpoints at wire speed, 100% free, no cluster changes.

For the Cloud Shell attachment to work, the NSG we tell Cloud Shell to use must be allowed to reach port 6443 on the `api_endpoint` NSG. That is the sole purpose of the new NSG and the new rule.
</background>
</context>

```hcl
# Cloud Shell Private Network → API endpoint (TCP 6443) — libera o operador a rodar
# `kubectl` e `terraform apply` direto do Cloud Shell enquanto ele estiver anexado ao NSG cloud_shell
resource "oci_core_network_security_group_security_rule" "api_endpoint_ingress_cloud_shell" {

      max = 6443
    }
  }
}
```

Do NOT add egress rules on the cloud_shell NSG. Cloud Shell Private Network works with ingress-only rules on the target NSG because Cloud Shell initiates the connection and NSGs are stateful (return traffic flows without an explicit egress rule on cloud_shell). This matches the pattern used for `bastion` NSG in the same file (it has no egress rule, only ingress from the operator CIDR).

Do NOT add the cloud_shell NSG to any subnet — Cloud Shell attaches the NSG to its own managed VNIC in the VCN at the moment the operator creates the Private Network connection in the console. The NSG just has to exist in the VCN.
